# Plan 3.1: EDD Foundation — Enrichment Schema, Golden Dataset & Deterministic Eval Harness

## Objective
Build the evaluation infrastructure BEFORE the LLM implementation. This is Evaluation-Driven Development: we define what "correct" looks like first, then build the processor that must satisfy it. All artifacts in this plan are deterministic (no LLM calls required).

## Context
- .gsd/SPEC.md
- .gsd/DECISIONS.md (ADR-003 — enrichment schema, 3-way category, 5-layer eval)
- src/scrapers/base_scraper.py (input schema: the 10-field unified item dict)
- src/clients/groq_client.py (existing Groq client — provides `analyze_json()`)

  <name>Create enrichment schema, validators & prompt template</name>
  <files>
    src/enrichment/__init__.py
    src/enrichment/schema.py
    src/enrichment/prompts.py
  </files>
  <action>
    Create a new `src/enrichment/` package with three files:

    **src/enrichment/__init__.py** — Empty init file.

    **src/enrichment/schema.py** — Enrichment schema definition and validation:
    1. Define `ENRICHMENT_FIELDS` dict mapping field names to their types and constraints:
       - `summary` (str, 50-500 chars)
       - `category` (str, enum: `battle-tested`, `new-upcoming`, `experimental`)
       - `effectiveness_score` (float, 0.0-1.0)
       - `relevance_score` (float, 0.0-1.0)
       - `tags` (list[str], from allowed set: `token-saving`, `multi-agent`, `prompting`, `frameworks`, `budget-control`, `workflow`, `configuration`, `debugging`, `cost-optimization`, `context-management`)
       - `enriched_at` (str, ISO timestamp — system-generated, not LLM)
       - `model_used` (str — system-generated, not LLM)
       - `enrichment_version` (str — prompt hash, system-generated)
    2. Define `VALID_CATEGORIES = {"battle-tested", "new-upcoming", "experimental"}`
    3. Define `VALID_TAGS` set with the allowed tag values
    4. Create `validate_enrichment(enrichment: dict) -> tuple[bool, list[str]]`:
       - Returns (is_valid, list_of_errors)
       - Checks: all required LLM fields present, category in enum, scores in 0-1 range, tags subset of VALID_TAGS, summary length within bounds
    5. Create `merge_enrichment(item: dict, enrichment: dict) -> dict`:
       - Takes a raw scraped item and enrichment result, returns a new dict with all original fields + enrichment fields
       - Adds system fields: `enriched_at`, `model_used`, `enrichment_version`
       - Does NOT mutate the input item

    **src/enrichment/prompts.py** — Prompt template and versioning:
    1. Define `SYSTEM_PROMPT` string — instructs the LLM to act as an AI strategy analyst evaluating agentic coding tools/techniques
    2. Define `USER_PROMPT_TEMPLATE` string — takes `{title}`, `{source}`, `{description}`, `{raw_content_truncated}`, `{tags}` placeholders. Instructs the LLM to return a JSON object with exactly: `summary`, `category`, `effectiveness_score`, `relevance_score`, `tags`. Include the allowed enum values for category and tags in the prompt itself.
    3. Create `build_prompt(item: dict, max_content_chars: int = 4000) -> str`:
       - Takes a scraped item dict, truncates `raw_content` to `max_content_chars`, fills the template
    4. Create `get_prompt_version() -> str`:
       - Returns SHA256 hash of `SYSTEM_PROMPT + USER_PROMPT_TEMPLATE` (first 12 hex chars)
       - This enables regression tracking: if prompt text changes, version changes

    IMPORTANT: Do NOT use generic placeholder prompts. Write the actual production-quality prompts that will be used. The prompt should be specific about agentic AI coding strategies and how to evaluate them.
  </action>
  <verify>
    python -m pytest tests/test_enrichment_schema.py -v
  </verify>
  <done>
    - `validate_enrichment()` correctly accepts valid enrichments and rejects invalid ones (wrong type, out-of-range scores, bad category, invalid tags)
    - `merge_enrichment()` produces a dict with all 10 original fields + 8 enrichment fields = 18 total
    - `build_prompt()` returns a string containing the item's title and truncated content
    - `get_prompt_version()` returns a stable 12-char hex hash that changes when prompt text changes
  </done>
